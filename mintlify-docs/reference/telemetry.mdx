SoulForge sends a single, anonymous "I'm running" ping when a session starts so
we can count active installs and see which versions and platforms are in use.
That's the entire purpose â€” no profiling, no tracking, no content.

<Note>
**Nothing personal or content-bearing ever leaves your machine.** No prompts,
no code, no file contents, no file paths, no cwd, no hostname, no username, no
API keys, no model IDs, and no IP address in the payload.
</Note>

### The anonymous id

`id` is a random UUID stored at `~/.soulforge/anon-id.json`. It exists so we can
tell "1 install pinged 50 times" apart from "50 installs" â€” i.e. count active
users, not raw requests. It is:

- **Random** â€” not derived from your machine, account, or anything identifiable.
  It identifies an *install*, not a person.
- **Stable** â€” the same id persists across runs, so distinct-id counts give
  exact daily/monthly active users, unique installs, and retention.
- **Local** â€” never linked to any other data. Delete the file (or reinstall)
  and you get a fresh id.

## How to opt out

Any **one** of these disables telemetry completely:

<CodeGroup>
```json config.json
{
  "telemetry": false
}
```

```bash Environment variable
# SoulForge-specific
export SOULFORGE_TELEMETRY=0

# Or the cross-tool industry standard (respected by many CLIs)
export DO_NOT_TRACK=1
```
</CodeGroup>

When disabled, no network request is made and no id is created or read.

## Design notes

- **Fail-safe.** The ping is fire-and-forget with a 1.5s timeout. It never
  blocks startup, never retries, and never surfaces an error â€” if you're offline
  or the endpoint is down, SoulForge behaves identically.
- **First-run notice.** The first time telemetry runs, a one-line notice is
  printed to stderr telling you it's on and how to turn it off.
- **Self-hostable.** Point the beacon at your own endpoint with
  `SOULFORGE_TELEMETRY_URL`.
- **No secrets shipped.** The client carries only a public, unauthenticated
  beacon URL â€” no API key, account id, or token. Reading the data requires a
  separate Cloudflare credential that never touches the client or the repo.
- **Abuse-resistant.** The endpoint is write-only into an analytics store (no
  database to read back or corrupt), with strict field allow-lists, per-IP rate
  limiting, and query-time dedup on the anonymous id.
- **Open.** The client lives in `src/core/telemetry.ts` and the receiving
  Cloudflare Worker is in `telemetry-worker/` â€” both in the public repo.
